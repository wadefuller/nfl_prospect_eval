# Draft Scout — Website

React + TypeScript + Vite + Tailwind CSS v4 frontend for the Draft Scout prospect model.

## Quick start

```bash
cd website
npm run dev      # dev server → http://localhost:5173/
npm run build    # production build → dist/
npm run preview  # preview production build
```

Node is installed via Homebrew: `/opt/homebrew/bin/node`

---

## Stack

| Layer | Choice |
|---|---|
| Framework | React 19 |
| Language | TypeScript (strict) |
| Build | Vite 8 |
| Styling | Tailwind CSS v4 |
| Routing | Hash-based (`#/`, `#/model`, `#/inspector`) |
| Data | Static JSON fetched at runtime from `public/data/` |

**Tailwind v4 syntax** — uses `@import "tailwindcss"` and `@theme {}` blocks in CSS, NOT `tailwind.config.js`. The Vite plugin is `@tailwindcss/vite` in `vite.config.ts`.

## Routes

| Hash | Component | What it shows |
|---|---|---|
| `#/` | `ProspectTable` | Sortable prospect board for a selected year |
| `#/model` | `ModelPage` | Model accuracy stats + methodology |
| `#/inspector` | `InspectorPage` | Cross-class player explorer |

Routing is handled with `useHashRoute()` in `App.tsx` — no React Router.

## Data flow

All data is static JSON in `public/data/`, generated by R scripts:

```
Rscript export_website_data.R    →  public/data/prospects/{year}.json
                                     public/data/comps/{player_id}.json
                                     public/data/meta.json
                                     public/data/model_performance.json
Rscript export_inspector_data.R  →  public/data/inspector/
```

**Never edit JSON files manually** — they are regenerated by the R pipeline. Edit the R export scripts instead.

### Data files

| File | Shape | Loaded by |
|---|---|---|
| `public/data/meta.json` | `{ availableYears, positions, lastUpdated, totalProspects }` | `App.tsx` on mount |
| `public/data/prospects/{year}.json` | `{ draftYear, prospects: Prospect[] }` | `App.tsx` on year change |
| `public/data/comps/{id}.json` | `{ prospectId, comps: ProspectComp[] }` | `App.tsx` on row expand |
| `public/data/model_performance.json` | `ModelPerformanceData` | `ModelPage.tsx` |
| `public/data/inspector/` | Inspector-specific JSONs | `InspectorPage.tsx` |

### Client-side derived fields
Two fields are computed in `App.tsx` after load:
- `draft_year` — tagged from `YearData.draftYear` (not stored in individual prospect objects)
- `ppg_if_hit` — `exp_ppg / p_made_it` (PPG conditional on making it; shown in detail panel)

## Type reference (`src/types.ts`)

Key types:

```typescript
Prospect         // A drafted WR or RB prospect with all model outputs
ProspectComp     // A single historical player comp (kNN match)
YearData         // { draftYear, prospects: Prospect[] }
Meta             // { availableYears, positions, lastUpdated, totalProspects }
ModelPerformanceData  // All model accuracy data for ModelPage
SortField        // Union of sortable column keys
```

`ppg_if_hit` is derived client-side and appended to `Prospect` on load — it is NOT in the JSON.

## Prospect fields decoded

| Field | Meaning |
|---|---|
| `exp_ppg` | Model's predicted half-PPR PPG (hurdle: p_made_it × E[ppg | hit]) |
| `p_made_it` | Probability of having at least one qualifying NFL season |
| `ppg_if_hit` | `exp_ppg / p_made_it` — PPG conditional on succeeding |
| `prospect_score` | Composite rank score (model + comp-stack blend) |
| `comp_weighted_ppg` | kNN comp-stack similarity-weighted historical PPG |
| `actual_ppg` | True career PPG (null for 2025–2026 prospects) |
| `actual_made_it` | True hit/bust flag (null for future classes) |
| `archetype` | Short string label, e.g. "Slot Speedster" |
| `bullish` / `bearish` | String arrays for the detail panel bullets |
| `has_wr_pbp` / `has_pbp` | 1 if play-by-play data was available (2014+) |

## ModelPage.tsx

The model accuracy page is written for two audiences:

1. **Plain-English section** ("How accurate is this model?") — stat tiles + plain cards + notable wins/misses
2. **Methodology section** — conceptual description of the modeling approach for ML practitioners, deliberately omits exact constants, file names, and variable names

When editing:
- `StatTiles` = "Avg miss", "Ranking quality", "Hit/miss accuracy", "Systematic skew"
- `HowItWorksCard` = five plain-English cards (The target, The classifier, The regressor, Training history, Calibration)
- `MethodologySection` / `MethodologySubsection` = structured text blocks
- Page title is "How accurate is this model?" — not "Methodology" or "About"

## Common tasks

**Add a new field to prospects:**
1. Add column to `export_website_data.R` output
2. Add field to `Prospect` interface in `src/types.ts`
3. Use in the relevant component

**Add a new draft year:**
1. Run the R pipeline through `07_score_all_classes.R`
2. Run `export_website_data.R` — it writes the new `{year}.json` and updates `meta.json`
3. No frontend changes needed — year picker reads from `meta.json`

**Change sort defaults for all-classes mode:**
- In `App.tsx`, `handleAllClassesToggle()` sets `sortField` and `sortDir` when toggling on

**Dev server doesn't pick up new JSON:**
- Vite serves `public/` statically; changes are picked up immediately on next page load

## Known issues / gotchas

- **2025–2026 prospects have null `actual_ppg`** — their NFL outcomes aren't yet resolved. Filter `actual_ppg !== null` before showing actual-vs-predicted comparisons.
- **`ppg_if_hit` can be misleadingly high for long-shot prospects** — `exp_ppg / p_made_it` amplifies for players with very low `p_made_it`. The UI shows `exp_ppg` as the primary sort column.
- **Comps are lazy-loaded** — `public/data/comps/{id}.json` is only fetched when a row is expanded. The `comps` state in `App.tsx` is a `Record<string, ProspectComp[]>` cache.
- **Inspector page is a separate data format** — JSON under `public/data/inspector/` has its own shape; see `InspectorPage.tsx` for the expected structure.
